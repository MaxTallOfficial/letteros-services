# Техническое задание: Letteros Short URL

## Контекст проекта

Нужно создать полноценную страницу сервиса сокращения ссылок для сайта letteros.com. Страница заменит текущую `letteros.com/shorturl/` — это полная переработка, а не редизайн. Текущая страница содержит только input-поле без оформления. Новая версия — это мини-лендинг с рабочим функционалом, бэкендом и описанием сервиса.

Сервис предназначен для незалогиненных пользователей (без авторизации).

**Репозиторий:** https://gitlab.letteros.com/maxtall/letteros-sevices
**Локальная папка:** `D:\ClaudeProjects\letteros-sevices`

Этот репозиторий — мультисервисный проект. В нём будет 3 страницы: `shorturl`, `shortcode` и `typograph`. Сейчас работаем **только над shorturl**. Для `shortcode` и `typograph` подготовь файловую структуру (пустые папки/роуты), но не создавай для них страницы и компоненты — для них будут отдельные ТЗ позже.

## Функционал бэкенда

> Этот раздел описывает техническую реализацию. Всё должно работать из одного репозитория.

### API: Сокращение ссылки
**POST** `/api/shorten`

Тело запроса:
```json
{
  "url": "https://example.com/very-long-url",
  "skipCheck": false,
  "customSlug": "my-link",
  "generateQr": true
}
```

Логика:
1. Валидация URL (формат, допустимые схемы — только http/https)
2. Если `skipCheck === false` — выполнить HTTP HEAD/GET запрос к URL и проверить, что он отвечает (таймаут — 10 секунд)
3. Если `customSlug` указан — проверить уникальность, валидировать формат (латиница, цифры, дефис, 3–30 символов)
4. Если `customSlug` не указан — сгенерировать случайный код (6–8 символов, буквенно-цифровой)
5. Сохранить в БД: оригинальный URL, короткий код, дата создания, дата истечения (90 дней от создания), IP-адрес создателя, счётчик переходов (0)
6. Вернуть результат с датой истечения

Ответ:
```json
{
  "shortUrl": "https://letteros.com/s/abc123",
  "code": "abc123",
  "expiresAt": "2026-06-24T00:00:00Z"
}
```

### API: Редирект
**GET** `/s/{code}`

Логика:
1. Найти запись по коду
2. Проверить, не истекла ли ссылка (90 дней)
3. Инкрементировать счётчик переходов
4. Выполнить 301 редирект на оригинальный URL
5. Если ссылка не найдена или истекла — показать понятную страницу ошибки

### Защита от злоупотреблений
- Rate limiting: максимум 10 сокращений в минуту с одного IP
- Базовая валидация URL (не допускать javascript:, data: и другие опасные схемы — только http и https)
- Максимальная длина URL: 2048 символов

### Срок жизни ссылок
- 90 дней для всех ссылок (пользователи не авторизованы)
- Периодическая очистка просроченных записей (предложи механизм, работающий из проекта)

## Детали реализации

### QR-код
- Используй npm-пакет `qrcode.react` (или аналог) — React-компонент, генерирует QR-код прямо в браузере
- QR-код генерируется только если пользователь включил соответствующий чекбокс
- Кнопка «Скачать QR-код» — конвертация canvas/SVG в PNG для скачивания

### Форма сокращения
- Валидация на клиенте перед отправкой (формат URL)
- Состояния: пустая форма → загрузка → результат / ошибка
- При ошибке проверки URL — понятное сообщение с предложением включить чекбокс «Не проверять ссылку»
- Тултип на иконке вопроса (?) — появляется при hover, на мобильных — при тапе

### Адаптивность
- Desktop, планшет, мобильные — страница должна корректно отображаться на всех разрешениях
- Следовать breakpoints из UIKit

### SEO
- Мета-теги: title, description, Open Graph
- Семантическая вёрстка (секции, заголовки, aria-атрибуты)

## Файловая структура проекта

Проект мультисервисный. Предусмотри структуру роутов под три сервиса, но реализуй только `shorturl`:

```
src/
├── app/
│   ├── shorturl/          ← РЕАЛИЗОВАТЬ (эта задача)
│   ├── shortcode/         ← только пустой роут-заглушка
│   ├── typograph/         ← только пустой роут-заглушка
│   ├── s/[code]/          ← редирект по короткой ссылке
│   └── api/
│       └── shorten/       ← API сокращения
├── components/
│   ├── shared/            ← общие компоненты (Header, Footer и т.д.)
│   └── shorturl/          ← компоненты страницы shorturl
└── ...
```

Это примерная структура — адаптируй под best practices Next.js, но сохрани идею: общие компоненты (шапка, подвал) — переиспользуемые, каждый сервис — в своей папке.

Для `shortcode` и `typograph` создай только пустые роуты с заглушкой (например, «Страница в разработке»). Никаких компонентов или логики для них не создавай.
